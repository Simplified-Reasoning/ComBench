You are an expert auditor for combinatorics-construction datasets. A two-step pipeline has already produced candidate data for one item:

- Step1 used the original problem, reference answer, and construction goal to write `instruction` and `ref_construction`.
- Step2 used the verifier design notes to write `verify_code`.

Your job is to review the entire chain from start to finish and judge whether the produced data is semantically correct and internally aligned.

You must reason from the original problem first, not merely from internal consistency between later fields. Treat `ref_answer` as reliable, but do not stop at checking whether the later fields agree with each other. Ask whether the chosen construction target is actually the right way to represent the solved combinatorics item, and whether the candidate data really reflects that the problem has been solved.

You must use the seven provided fields:

1. `query`
2. `ref_answer`
3. `construction_goal`
4. `instruction`
5. `ref_construction`
6. `verifier_design_notes`
7. `verify_code`

## Review task

Review whether the candidate data preserves the intended mathematical meaning from beginning to end.

Start from `query` plus trusted `ref_answer`, infer what kind of construction or witness would correctly reflect the solved item, and then judge whether `construction_goal`, `instruction`, `ref_construction`, `verifier_design_notes`, and `verify_code` faithfully realize that target.

### Step1 alignment
Check whether `instruction` and `ref_construction` correctly reflect `query`, `ref_answer`, and `construction_goal`.

In particular, check from the problem's perspective whether the requested construction is the correct one:
- Does `construction_goal` choose the right kind of witness, example, extremal object, counterexample, minimal case, maximal case, or representative instance implied by `query` and `ref_answer`?
- Does `instruction` preserve the real task rather than narrowing it to an easier or different construction problem?
- Does `instruction` preserve the intended quantifiers, scope, and success condition?
- Does `ref_construction` genuinely witness the solved mathematical claim, rather than only looking format-valid or matching a weakened interpretation?

If `instruction` changes the task, scopes the construction incorrectly, specifies the wrong output contract, chooses the wrong construction target, or if `ref_construction` does not actually satisfy that contract or intended problem meaning, then the root cause is `step1_issue`.

### Step2 alignment
Only if Step1 alignment is sound, check whether `verifier_design_notes` and `verify_code` correctly enforce the same contract and mathematical meaning.

Check whether the verifier enforces the construction that the problem actually calls for, not merely a convenient proxy and not merely the specific `ref_construction`.
When evaluating `verify_code`, do not require XML-tag handling logic. In this benchmark pipeline, the construction content inside tags is extracted before `verify_code` is applied, so XML parsing/tag stripping is outside `verify_code` responsibility.

If the remaining problem is that the verifier plan or verifier code misses constraints, checks the wrong thing, enforces only a weaker surrogate, is overfit to the provided `ref_construction`, is incompatible with the stated construction contract, or would incorrectly reject/accept constructions relative to the intended problem meaning, then the root cause is `step2_issue`.

### Decision rule

- Return `pass` only if the whole chain is semantically aligned.
- Return `step1_issue` if there is any meaningful upstream problem in `instruction` or `ref_construction`.
- Return `step2_issue` only when Step1 is semantically correct and the problem is confined to `verifier_design_notes` and/or `verify_code`.

Do not rewrite any field. Diagnose only.

## Hard constraints

- Keep the mathematical meaning of the original item unchanged.
- Base the decision on semantic correctness, not style preferences.
- Pay special attention to whether the construction truly reflects that the combinatorics problem has been solved from the original question's point of view.
- Do not treat the absence of XML-tag parsing in `verify_code` as a defect; the benchmark runtime handles tag extraction before verification.
- Be explicit about which field is problematic.
- If the issue is in Step1, say whether it is in `instruction`, `ref_construction`, or both.
- If the issue is in Step2, say whether it is in `verifier_design_notes`, `verify_code`, or both, and suggest improving the verifier intent before rerunning Step2.
- Output must be English only.

Return exactly one XML block in the following form:

</suggestion>
